go()

Our go() method ties everything together. This method takes an object, specifically, your test data structure and turns that structure in to specific actions and asserts that are executed in the browser.

The go() method aims to validate the data structure you pass as much as possible to ensure that it can do what is required. If the data structure is missing something, go() will probably throw an error and a stacktrace. This isn't really perfect yet, but we're working to make it better. Let us know where we can improve.

A good data structure

Let's break down what a good data structure looks like. As a reference, let's look at our DuckDuckGo test. 

'use strict';


var drive = require('datadriver');


drive.go({


  "config": {
    "description": "Duck Duck Go Test",
    "suites": [
      {
        "description": "Search for DataDriver.io",
        "tests": ["duck-duck-go"],
        "viewports": { small: true }
      }
    ]
  },


  "duck-duck-go": {
    "steps": [
      {
        "selector": "#search_form_input_homepage",
        "execute": [{ "action": "type", "text": "Datadriver.io" }]
      },
      {
        "selector": "#search_button_homepage",
        "execute": [{ "action": "click" }],
      },
      {
        "verify": [
          {
            "method": "verifyPageTitle",
            "asserts": { "title": "Datadriver.io at DuckDuckGo" }
          }
        ]
      }
    ]
  }


});

As you can see from the code example above, the drive.go() method takes an object with a particular structure. In this structure we have two keys, one is "config" (and it always needs to be named "config") and the other is called "duck-duck-go", which we can call anything we want, as long as we reference it correctly from the "config.tests" array.

Let's talk about the "config"

Here is the code we're talking about for reference:

"config": {
  "description": "Duck Duck Go Test",
  "suites": [
    {
      "description": "Search for DataDriver.io",
      "tests": ["duck-duck-go"],
      "viewports": { small: true }
    }
  ]
}

As mentioned before, the "config" object must always be named "config". This object sets up our BDD interface, allowing our tests to execute in a logical manner. It's important to know that we don't have hooks in to everything the interface supports, as the go() method handles all of this for you, but in short, the BDD interface allows for the output you've been seeing in our examples.

If you look at the test itself and the output you're given, you can see that there's a pretty good match up. So let's walk through the "config" object. 

  • "config": {
  "description": "Duck Duck Go Test",
  "suites": [
    {
      "description": "Search for DataDriver.io",
      "tests": ["duck-duck-go"],
      "viewports": { small: true }
    }
  ]
}
  • "description": This is the description of our test file. If I'm testing various elements on a page, I'll give the description something like "DuckDuckGo Homepage Elements". If it's a step-by-step flow, I'll give the description something like "Sign up flow".
  • "url": Not shown in this example, but you can see how we can use it on the getBaseUrl() page. The url field doesn't take an absolute url, only the the path you need in addition to the url returned by getBaseUrl. So, for example, if I passed in --url=http://duckduckgo.com from the command line and changed my "url" field to be "url": "about", the end result would be http://duckduckgo.com/about.
  • "suites": The "suites" key should always take an array of objects. Each object, marked by their { } defines a group of tests that should be run together. Each suite object has three keys we need to define.
    • "description": This is the description of your test suite. Again, try to give it a description that says these tests go together.
    • "tests": The "tests" also receives an array. This allows you to choose from multiple test objects and arbitrarily string them together. For example, we could include another test object that verifies a logo on the DuckDuckGo about page and string it together in our suite. Let's take a look at what that could look like.

        "viewports": { small: true }
      }
    ]
  },

    "steps": [
      {
        "selector": "#search_form_input_homepage",
        "execute": [{ "action": "type", "text": "Datadriver.io" }]

    • "viewports": This key takes an object, which corresponds to any viewports that you have defined in the gruntfile. Please note you need at least one viewport defined in the gruntfile.   "medium": true

Let's talk about test objects

The following is a test object.

"duck-duck-go": {
  "steps": [
    {
      "selector": "#search_form_input_homepage",
      "execute": [{ "action": "type", "text": "Datadriver.io" }]
    },
    {
      "selector": "#search_button_homepage",
      "execute": [{ "action": "click" }],
    },
    {
      "verify": [
        {
          "method": "verifyPageTitle",
          "asserts": { "title": "Datadriver.io at DuckDuckGo" }
        }
      ]
    }
  ]
}

The key for this test object is "duck-duck-go", but you can name it anything you want, as long as you reference it inside of the "tests" array in the "config". Inside the test object you will immediately encounter a "steps" array, which just takes a list of objects to be executed in sequential order.

Each step object can have the following keys:

  • "selector": This sets the context of actions or asserts inside of the step. If you look at the above example, our first step searches for an element on the page with an id of search_form_input_homepage (we're using CSS selectors here, but you can also use XPath). Therefore, the execute action that follows will be executed on that selector. You don't have to pass a selector, in which case the default selector is window (not a valid CSS selector).
  • "execute": The execute key takes an array, full of action objects. You can specify multiple actions to be executed per step. More details are available under "Execute" in the sidebar.
  • "verify": The verify key takes an array full of assert objects. Each assert object takes a "method" key and an "asserts" object of key value pairs that you want to verify to be true. We use the Chai.js assertion library if you feel like changing the assert-mode of an assert object. More details under assert modes. For examples of how to use a particular verification step, visit any of the verify links in the sidebar.
